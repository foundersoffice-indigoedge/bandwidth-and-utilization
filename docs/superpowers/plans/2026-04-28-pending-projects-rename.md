# Pending Projects Rename + Admin Workflow Removal — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Stop calling user-added projects "ad hoc," reshape the underlying DB table as a one-way outbox drained by an external automation, rip out the in-app admin link/connect workflow, and reframe the form remarks field around non-project threads.

**Architecture:** A fellow adds a new project via the submit form → a row is written to `pending_projects` with `status='pending'` → an external automation reads the table, creates the corresponding Airtable record, sets `status='finished'`, then deletes the row. The internal admin "Link to Airtable" UI and its supporting API routes are removed. The form's "Added by you / teammates" section becomes "New projects." The free-form remarks textarea is reframed for sector scoping, outreach, and other non-project threads (projects already have a structured path).

**Tech Stack:** Next.js 16 App Router, Drizzle ORM 0.31 with Postgres (Neon), Vitest, Slack incoming webhook.

**Out of scope:** The external automation itself (separate workstream). This plan only shapes the DB contract it'll consume.

## Decisions Locked In

These were settled with the user up-front and should not be re-litigated during execution:

| Decision | Choice |
|---|---|
| DB table name | Rename `ad_hoc_projects` → `pending_projects` |
| Status enum | `pending` (default) → automation flips to `finished` → automation deletes the row |
| Section heading in form | "New projects" |
| Slack notification | Keep firing, change title to `:new: *New Mandate*` / `*New DDE*` / `*New Pitch*`; body unchanged |
| `conflicts.is_ad_hoc` | Drop (set on insert, never read) |
| `linked_airtable_record_id` & `linked_at` | Drop (only used by the removed link workflow) |
| Project record id prefix | `adhoc_<uuid>` → `pending_<uuid>` (migrate existing rows in submissions & conflicts) |
| TS field on `Project` interface | `isAdHoc?: boolean` → `isNew?: boolean` |
| Remarks placeholder | "any sector scoping, outreach or other threads you are working on" |
| `existingAdHocId` branch in add-project route | Delete (dead — only the removed admin flow ever set it) |
