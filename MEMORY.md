## Learnings

### Airtable Field Names
- Field naming is inconsistent across tables. Mandates use "Mandate VP / AVP 1", DDEs use "DDE VP / AVP", Pitches use "Pitch VP / AVP". All handled via a per-table config map in `app/src/lib/airtable/config.ts`.
- DDEs have 1 VP/AVP slot and 1 Associate slot. Mandates and Pitches have 2 of each.
- All team role fields are `multipleRecordLinks` with `prefersSingleRecordLink: true` — they return arrays of record IDs but typically contain 0 or 1 element.
- Fellows List table field names don't follow a simple pattern: `Name of Fellow` (not `Name`), `Email ID of Fellow` (not `Email`), `Designation of Fellow` (not `Designation`). Verified via Rube MCP against live Airtable schema on 2026-04-07. `Capacity [MEU]` is present in the schema but the app no longer reads it (hours ÷ 84 is the sole capacity signal since April 17, MEU path fully removed April 21).

### Architecture
- Resend SDK creates a client at module level. Any module that imports email.ts transitively will crash in tests without RESEND_API_KEY. The `isCycleMonday` function was extracted to `schedule.ts` and re-exported from `cycle.ts` to keep pure functions testable without env vars. Same pattern should be used for any future pure functions in modules that import SDK clients.
- DB client uses a lazy Proxy pattern (`lib/db/index.ts`) so importing the module doesn't crash without DATABASE_URL. Only actual queries trigger the connection.
- Dashboard monthly rows and overview grid show **averages** across all cycle snapshots in a month (typically 2 biweekly), not the latest snapshot. Weekly rows still show exact per-cycle data. Load tag for monthly average is recomputed client-side via a duplicate `getLoadTag` function in `DashboardView.tsx`.
- Week-to-cycle matching uses **overlap** (any overlap = match), not midpoint. Search runs against all fellow snapshots, not just the current month's, because a cycle starting late in the previous month can cover early days of the next month.
- Dashboard has two data sources: `snapshots` table for finalized historical data (monthly grid), and `submissions` table for the active cycle ("Current Cycle" live section). The live section only appears when a cycle is in `collecting` status and disappears after finalization. Only one active cycle exists at a time.
- Dashboard sorts by designation hierarchy (VP > AVP > Associate 3 > 2 > 1 > Analyst), then alphabetical within each tier. Applied to live table, pending list, and overview grid via `DESIGNATION_RANK` map in `DashboardView.tsx`. Live table has a toggle to switch between Designation and Load (utilization % high to low) sorting.
- `WORKING_DAYS_PER_WEEK` constant in `scoring.ts` centralizes the per-day to per-week conversion factor (currently 6, reflecting IE's 6-day work week). All conversion paths import this constant instead of hardcoding the multiplier, so future changes are a one-line edit.
- Conflict resolution sends a threaded follow-up email (`In-Reply-To` / `References` headers) so unresolved conflicts are visible in Gmail as threads without a reply. The `emailMessageId` column on conflicts stores the Resend message ID from the original email. Conflicts created before this feature won't thread (no stored ID).
- Pseudo-snapshot injection: to blend in-progress cycle data into the Monthly view, the page synthesizes a snapshot row per submitted fellow with `id: pseudo-<cycleId>-<fellowId>` and `snapshotDate: activeCycle.startDate`. Monthly averaging logic treats them identically to real snapshots — zero code change in the aggregator.
- Two-source Latest Cycle: `getLiveCycleData()` tries active `collecting` cycle first, falls back to `getLatestFinalizedCycleData()` which pulls the most recent `complete` cycle's snapshots. Discriminated by `status: 'collecting' | 'complete'` so the UI shows "Live" vs "Finalized" badge.
- Ad-hoc project similarity matching uses dice coefficient on character bigrams (`lib/similarity.ts`). Fast enough to rank all active Airtable projects of the same type on a single request. No fuzzy-matching lib needed.
- Tier grouping: `getTier(designation)` maps VP/AVP direct, any "Associate *" → Associate, Analyst → Analyst, unknown → Analyst. `TIER_ORDER` drives consistent sort. `lib/tiers.ts` is the single source.
- `getCycleEndDate()` in `schedule.ts` is the single source of truth for cycle end dates. End = day before the next cycle's Monday. For cycles that start before the weekly anchor (`2026-04-27`), "next" is the anchor itself — that's what handles the biweekly→weekly transition cleanly (Apr 17 cycle ends Apr 26, not Apr 23). Every display site — email subject, dashboard "Latest Cycle" header, Slack reminder — routes through this one function. Adding a new place that shows a cycle range? Import `getCycleEndDate`, don't roll your own `+6`.
- Weekly drill-down rule changed: `findSnapshotForWeek` now returns the snapshot whose cycle start falls INSIDE the week boundaries (was overlap + latest-wins). Side effect: weekly cells stay empty until that week's cycle is collected, which matches the weekly-collection mental model. Historical biweekly snapshots only appear in the week their start Monday fell in — they no longer duplicate into the empty second week they used to cover.
- Per-project conflict surfacing: `ProjectBreakdownItem.hasConflict` is derived server-side in `dashboard/page.tsx` from the set of pending-conflict project record IDs, then consumed by the shared `ProjectBreakdownTable`. Historical snapshots store breakdown rows without this field, so the badge gates on `b.hasConflict === true` — no branching needed between Live vs historical render paths, finalized cycles just don't light it up.
- **Latest Cycle drill-down has two render sources, both now carry remarks.** `getLiveCycleData` (active `collecting` cycle) reads remarks off any non-null self-report submission for the fellow — they're duplicated across self-report rows by `/api/submit:130`, so picking the first non-empty one is sufficient. `getLatestFinalizedCycleData` (finalized fallback) joins the snapshot list with a parallel `submissions WHERE cycleId = X AND isSelfReport = true` query and builds a `Map<fellowRecordId, remarks>`. Both paths produce the same `LiveFellowData` shape so `LiveDrillDown` doesn't branch.
- **Monthly Report scroll required per-tier scroll containers.** The original layout had `overflow-x-auto` on the outer wrapper but `overflow-hidden` on each tier card (for the rounded corners), which meant the table got clipped before its overflow could propagate to the scroll container. Fix: keep `overflow-hidden` on the tier card (preserves the rounded look), wrap the `<table>` itself in a new `<div className="overflow-x-auto">`, and switch the table from `w-full` to `min-w-full` so it's free to grow when 12 month columns × `min-w-[100px]` exceed the viewport. The existing sticky Fellow column (`sticky left-0`) anchors correctly to the new inner scroll viewport without further changes.
- **useEffect is the wrong tool for deriving state from props.** The first cut of the ad-hoc reconciler used `useState` + a `useEffect` that synced entries when `projects` changed. The effect runs AFTER the render pass that introduced the new prop, so the first render with a newly-added ad-hoc project accessed `entries[newKey]` as `undefined` inside `HoursInput` and crashed. Correct pattern: keep a `userInput` state for typed values, then `useMemo` a derived `entries` map during render by merging `userInput` with defaults synthesized from the current `projects`. Pure function `deriveEntries()` lives in `form-entries.ts` and is exhaustively unit-tested (15 tests covering mandate/dde/pitch, ad-hoc, VP vs non-VP, user-input preservation). The `deriveEntries` file is decoupled from React so the tests never spin up a DOM.
- No custom `error.tsx` in the submit route, so any uncaught render throw bubbles to the Next.js default handler, which on Vercel renders as the browser's generic "This page couldn't load" shell. If we ever want friendlier in-form recovery, add `src/app/submit/[token]/error.tsx`.
- `deriveEntries()` supports an optional `initialEntries` param so ad-hoc projects that were added mid-session come back pre-filled on form reload. Precedence order inside the function: `userInput[key]` → `initialEntries[key]` → empty default. Page component reads existing submissions where `projectRecordId LIKE 'adhoc_%'` for the current fellow/cycle and passes them through. Five additional unit tests cover the initialEntries behavior; full suite sits at 112/112.
- `/api/submit` now dispatches on `projectRecordId.startsWith('adhoc_')`: regular projects insert new rows (unchanged), ad-hoc rows UPDATE the existing submission created at add-time. Without this dispatch, ad-hoc entries were silently skipped at line 48's `if (!project) continue` — the UI showed values but the server didn't save edits. This is why the read-only hack existed briefly; the proper fix is the update path. Remarks still flow into ad-hoc self-reports on update (consistent with regular self-report remark handling).
- Badge label in the submit form next to ad-hoc project names is the project type (Mandate / DDE / Pitch), not the generic word "ad-hoc". The type is already conveyed by the section header ("Added by you / teammates"); the badge slot communicates type-of-project, matching how regular projects show their stage.
- Fellow x Project timeline reads `submissions INNER JOIN cycles` alone (no `conflicts` join) because conflict resolution UPDATE-s both the VP and associate submission rows with the resolved value (`api/resolve/route.ts:56-77`). Post-resolution, submissions is single source of truth. Query filters with `or(and(fellowRecordId=X, isSelfReport=true), and(targetFellowId=X, isSelfReport=false))` — union of a fellow's own self-reports and VP projections aimed at them.
- Pure-function core for testability: `buildTimeline`, `listProjectsForFellow`, `iyOf` live in `lib/fellow-project-timeline.ts` with zero Next/DB/React imports. 18 unit tests cover IY boundaries, self-vs-projection dedup (array-order independent), IY filtering, alphabetical sort, multi-IY selection. The API route is a thin adapter that loads rows + calls these pure functions + returns JSON. Same pattern as `form-entries.ts`: if a piece of logic can be a pure function, make it one and test it exhaustively before touching the UI.
- Three-mode API (`GET /api/dashboard/fellow-project?mode=bootstrap|projects|timeline`): cascading picker pattern. Bootstrap returns all fellows + all IYs the system has seen. `projects` takes fellow + IYs (comma-separated), returns that fellow's projects within those IYs. `timeline` takes fellow + project, returns the sorted point series. Keeps each request cheap — no pre-loading everyone's entire history.
- IY (IndigoEdge fiscal year) convention: July–June. `iyOf(dateStr)` returns `year + 1` for Jul–Dec dates, `year` for Jan–Jun. IY2026 = July 2025 through June 2026. The dashboard's existing `getIyRange(iy)` uses the same convention — both should stay in sync if the fiscal calendar changes.

### Testing
- Full self-test validated the complete workflow (cycle start through dashboard rendering). Airtable writeback format differs from historical manual entries (plain text vs. markdown headers), making test data easy to identify but also means the system's output will eventually replace manual data.
- `TEST_EMAIL_OVERRIDE` pattern works well for safe production testing. Set the env var, run the test, remove the var. No code changes needed.
- Cycle finalization writes bandwidth to ALL projects a fellow has self-reports for (not just ones the VP reported on). When testing with a fellow who has many mandates, expect many Airtable records to be updated. Clean them all up.
- Seed script (`app/seed-test-data.mjs`) uses `rec_test_` prefixed fellow IDs for safe cleanup. `node seed-test-data.mjs --clean` deletes all test data via `DELETE WHERE fellow_record_id LIKE 'rec_test_%'`. Script is idempotent (cleans before re-seeding).

### Email Delivery
- Resend Node SDK (v2+) returns `{ data, error }` instead of throwing on failure. Must destructure and check `error` explicitly, or failures are silently swallowed. The `sendEmail()` wrapper in `email.ts` handles this.
- Resend rate-limits at ~2 emails/second. Sequential sends without delays will get partially dropped. A 500ms `sleep()` between sends in `cycle.ts` and `send-reminders/route.ts` keeps us under the limit. Adds ~16 seconds to a 33-person cycle start, which is fine for a biweekly function.

### Deployment
- `vercel.ts` (TypeScript config via `@vercel/config/v1`) caused silent deploy failures with empty build logs. Replaced with standard `vercel.json`. Stick with `vercel.json` for now.
- Vercel `git connect` can't parse SSH host aliases (e.g., `git@github-ie-central:`). Temporarily set remote to `https://github.com/...` URL, connect, then restore SSH remote.
- `vercel env add` via CLI reads from stdin. Use `printf` (not `echo`) to avoid trailing newline/whitespace, which Vercel rejects.
- Drizzle schema drift: `drizzle-kit push` syncs schema.ts to the DB but doesn't generate migration files. After Phase 1 of v2, prod was correct but `drizzle/` was 5 commits behind schema.ts. Run `drizzle-kit generate` alongside any schema change and commit the SQL file so the on-disk migrations stay a faithful diff record. Generated 0001_v2_schema.sql retroactively on 2026-04-20 — no-op against prod but paired with schema.ts going forward.
- Env values pulled via `vercel env pull` come back wrapped in double quotes with a literal `\n` trailing many of them (an artefact of how they were uploaded). Local test scripts need to strip both before use — a quick regex pass (`key="?value"?` → `key=value`, then trim trailing `\n`) is enough. The production runtime doesn't see this because Vercel normalizes at inject time; it only bites when you source the pulled file locally.
- Dropping a Drizzle-managed column when the new app code still reads it will crash prod. Safe order: (1) ship code that stops reading the column, (2) ship code that stops writing it, (3) update schema.ts, (4) `drizzle-kit generate`, (5) apply the migration. We did 1+2+3 in the same commit this round because the column had been write-only for days, but the rule still holds: never let the column drop land before the last reader.

### Feature Flags & Rollout

- `DISABLE_AIRTABLE_WRITEBACK=true` in prod pauses all writeback to Airtable. Code still computes what it would write. Re-enable by setting the var to anything else (or removing it). Pattern: single env flag checked at the writeback callsite, no code branches needed elsewhere.
- Weekly vs biweekly cadence is a single-constant change (`CYCLE_ANCHOR` + `WEEKS_PER_CYCLE` in `schedule.ts`). Reference-date schedule math absorbs the switch — no data migration needed when changing cadence.
