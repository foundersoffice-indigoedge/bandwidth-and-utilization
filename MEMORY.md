## Learnings

### Airtable Field Names
- Field naming is inconsistent across tables. Mandates use "Mandate VP / AVP 1", DDEs use "DDE VP / AVP", Pitches use "Pitch VP / AVP". All handled via a per-table config map in `app/src/lib/airtable/config.ts`.
- DDEs have 1 VP/AVP slot and 1 Associate slot. Mandates and Pitches have 2 of each.
- All team role fields are `multipleRecordLinks` with `prefersSingleRecordLink: true` — they return arrays of record IDs but typically contain 0 or 1 element.
- Fellows List table field names don't follow a simple pattern: `Name of Fellow` (not `Name`), `Email ID of Fellow` (not `Email`), `Designation of Fellow` (not `Designation`). Verified via Rube MCP against live Airtable schema on 2026-04-07. `Capacity [MEU]` is present in the schema but the app no longer reads it (hours ÷ 84 is the sole capacity signal since April 17, MEU path fully removed April 21).

### Architecture
- Resend SDK creates a client at module level. Any module that imports email.ts transitively will crash in tests without RESEND_API_KEY. The `isCycleMonday` function was extracted to `schedule.ts` and re-exported from `cycle.ts` to keep pure functions testable without env vars. Same pattern should be used for any future pure functions in modules that import SDK clients. (Reinforced 2026-06-26: the peer-email time-trigger's pure logic lives in `peer-email-schedule.ts` and `signoff-scope.ts` — separate from `cycle.ts`, which pulls in Resend — so the IST day/checkpoint and expected-director logic are unit-tested without env.)
- Peer-email idempotency uses an **atomic claim**, not read-then-write: `UPDATE cycles SET peer_emails_sent=true WHERE id=? AND peer_emails_sent=false RETURNING id` — only the invocation that gets a row back proceeds to send. Four Tue/Wed cron checkpoints hit the same cycle, so a plain read-check-then-write would double-send under cron overlap or Vercel retry. On a total send failure the claim is released (flag back to `false`) so a later checkpoint retries; partial failures keep the claim to avoid re-emailing recipients who already received it.
- DB client uses a lazy Proxy pattern (`lib/db/index.ts`) so importing the module doesn't crash without DATABASE_URL. Only actual queries trigger the connection.
- Dashboard monthly rows and overview grid show **averages** across all cycle snapshots in a month (typically 2 biweekly), not the latest snapshot. Weekly rows still show exact per-cycle data. Load tag for monthly average is computed client-side via the canonical rule-backed `getLoadTag` imported from `utilization.ts` (the former duplicate function in `DashboardView.tsx` was removed in Phase D, 2026-06-15).
- Week-to-cycle matching uses **overlap** (any overlap = match), not midpoint. Search runs against all fellow snapshots, not just the current month's, because a cycle starting late in the previous month can cover early days of the next month.
- Dashboard has two data sources: `snapshots` table for finalized historical data (monthly grid), and `submissions` table for the active cycle ("Current Cycle" live section). The live section only appears when a cycle is in `collecting` status and disappears after finalization. Only one active cycle exists at a time.
- Dashboard sorts by designation hierarchy (VP > AVP > Associate 3 > 2 > 1 > Analyst), then alphabetical within each tier. Applied to live table, pending list, and overview grid via `DESIGNATION_RANK` map in `DashboardView.tsx`. Live table has a toggle to switch between Designation and Load (utilization % high to low) sorting.
- `WORKING_DAYS_PER_WEEK` in `scoring.ts` centralizes the per-day to per-week conversion factor (currently 6, IE's 6-day work week). Since Phase D (2026-06-15) it resolves from the governed rule `utilization-mis.calc.working-days-per-week` rather than a hardcoded literal; all conversion paths import the constant, so a change is one Notion edit. The transition doc's "inlined 6/84" copies (signoff.ts, add-project, DashboardView) were all rewired to the constant in the same pass.
- Conflict resolution sends a threaded follow-up email (`In-Reply-To` / `References` headers) so unresolved conflicts are visible in Gmail as threads without a reply. The `emailMessageId` column on conflicts stores the Resend message ID from the original email. Conflicts created before this feature won't thread (no stored ID).
- Pseudo-snapshot injection: to blend in-progress cycle data into the Monthly view, the page synthesizes a snapshot row per submitted fellow with `id: pseudo-<cycleId>-<fellowId>` and `snapshotDate: activeCycle.startDate`. Monthly averaging logic treats them identically to real snapshots — zero code change in the aggregator.
- Two-source Latest Cycle: `getLiveCycleData()` tries active `collecting` cycle first, falls back to `getLatestFinalizedCycleData()` which pulls the most recent `complete` cycle's snapshots. Discriminated by `status: 'collecting' | 'complete'` so the UI shows "Live" vs "Finalized" badge.
- `lib/similarity.ts` (dice coefficient on character bigrams) is now orphaned after the 2026-04-28 admin link-workflow removal. Kept in place because it's a small pure function with zero deps; if fuzzy matching is ever needed elsewhere, this is the utility. Was originally written to rank Airtable project-name candidates against pending-project names.
- Tier grouping: `getTier(designation)` maps VP/AVP direct, any "Associate *" → Associate, Analyst → Analyst, unknown → Analyst. `TIER_ORDER` drives consistent sort. `lib/tiers.ts` is the single source.
- **`resolveProjectRole` returns `'associate'` as a catch-all — for anyone not in a senior slot — so never treat that as proof of an associate placement.** The peer email tagged VP/AVPs "Performing Associate role" whenever their project was missing from the fetch (deleted / inactive-stage / `pending_`) or they'd been reassigned off the team mid-cycle; both surface as role `'associate'` by elimination, and combined with `isVpOrAvp(designation)` produced a false pill. Any label/behaviour meaning "this fellow is an associate here" must check **affirmative membership** (`proj.associateIds.includes(fellowRecordId)`), never the absence of a senior role. The durable fix has two layers: drop orphaned rows entirely (`filterLiveSelfReports`, live cycle only) and, defensively, gate the pill on `associateIds` membership. Two mislabel triggers were confirmed against live data (Cent.ai DDE completed-stage + Foxtale deleted → project absent from fetch; PlatinumRx Pitch → Murali swapped off the team ~5h after the cycle fired).
- Role/authority in the bandwidth flow is determined by the fellow's **designation** (`isVpOrAvp(designation)`), not by which Airtable column they occupy. The form's reporting mode (`submit/[token]/page.tsx`), the VP↔Associate conflict cross-check (`api/submit/route.ts`), and the director-flag resolver (`director-flag.ts`) all branch on designation. So a VP/AVP can sit in any project column (including the Director slot, when they lead a DDE) and still be treated as a VP. Corollary: `getProjectsForFellow` only decides *which* projects appear on a fellow's form, never the role they play on them — keep role logic designation-driven and you don't need to normalize the column arrays.
- `getCycleEndDate()` in `schedule.ts` is the single source of truth for cycle end dates. End = day before the next cycle's Monday. For cycles that start before the weekly anchor (`2026-04-27`), "next" is the anchor itself — that's what handles the biweekly→weekly transition cleanly (Apr 17 cycle ends Apr 26, not Apr 23). Every display site — email subject, dashboard "Latest Cycle" header, Slack reminder — routes through this one function. Adding a new place that shows a cycle range? Import `getCycleEndDate`, don't roll your own `+6`.
- Weekly drill-down rule changed: `findSnapshotForWeek` now returns the snapshot whose cycle start falls INSIDE the week boundaries (was overlap + latest-wins). Side effect: weekly cells stay empty until that week's cycle is collected, which matches the weekly-collection mental model. Historical biweekly snapshots only appear in the week their start Monday fell in — they no longer duplicate into the empty second week they used to cover.
- Per-project conflict surfacing: `ProjectBreakdownItem.hasConflict` is derived server-side in `dashboard/page.tsx` from the set of pending-conflict project record IDs, then consumed by the shared `ProjectBreakdownTable`. Historical snapshots store breakdown rows without this field, so the badge gates on `b.hasConflict === true` — no branching needed between Live vs historical render paths, finalized cycles just don't light it up.
- **Latest Cycle drill-down has two render sources, both now carry remarks.** `getLiveCycleData` (active `collecting` cycle) reads remarks off any non-null self-report submission for the fellow — they're duplicated across self-report rows by `/api/submit:130`, so picking the first non-empty one is sufficient. `getLatestFinalizedCycleData` (finalized fallback) joins the snapshot list with a parallel `submissions WHERE cycleId = X AND isSelfReport = true` query and builds a `Map<fellowRecordId, remarks>`. Both paths produce the same `LiveFellowData` shape so `LiveDrillDown` doesn't branch.
- **Live reconciliation has 2 inputs with different jobs.** Raw self-reports decide whether a fellow submitted and carry remarks. Reconciled self-reports decide current hours and active project breakdown. Keep that split in `buildLiveDashboardFellow()` so a fellow whose only project was paused remains visible at 0 load with their original remark.
- **Monthly Report scroll required per-tier scroll containers.** The original layout had `overflow-x-auto` on the outer wrapper but `overflow-hidden` on each tier card (for the rounded corners), which meant the table got clipped before its overflow could propagate to the scroll container. Fix: keep `overflow-hidden` on the tier card (preserves the rounded look), wrap the `<table>` itself in a new `<div className="overflow-x-auto">`, and switch the table from `w-full` to `min-w-full` so it's free to grow when 12 month columns × `min-w-[100px]` exceed the viewport. The existing sticky Fellow column (`sticky left-0`) anchors correctly to the new inner scroll viewport without further changes.
- **useEffect is the wrong tool for deriving state from props.** The first cut of the ad-hoc reconciler used `useState` + a `useEffect` that synced entries when `projects` changed. The effect runs AFTER the render pass that introduced the new prop, so the first render with a newly-added ad-hoc project accessed `entries[newKey]` as `undefined` inside `HoursInput` and crashed. Correct pattern: keep a `userInput` state for typed values, then `useMemo` a derived `entries` map during render by merging `userInput` with defaults synthesized from the current `projects`. Pure function `deriveEntries()` lives in `form-entries.ts` and is exhaustively unit-tested (15 tests covering mandate/dde/pitch, ad-hoc, VP vs non-VP, user-input preservation). The `deriveEntries` file is decoupled from React so the tests never spin up a DOM.
- No custom `error.tsx` in the submit route, so any uncaught render throw bubbles to the Next.js default handler, which on Vercel renders as the browser's generic "This page couldn't load" shell. If we ever want friendlier in-form recovery, add `src/app/submit/[token]/error.tsx`.
- `deriveEntries()` supports an optional `initialEntries` param so ad-hoc projects that were added mid-session come back pre-filled on form reload. Precedence order inside the function: `userInput[key]` → `initialEntries[key]` → empty default. Page component reads existing submissions where `projectRecordId LIKE 'adhoc_%'` for the current fellow/cycle and passes them through. Five additional unit tests cover the initialEntries behavior; full suite sits at 112/112.
- `/api/submit` now dispatches on `projectRecordId.startsWith('adhoc_')`: regular projects insert new rows (unchanged), ad-hoc rows UPDATE the existing submission created at add-time. Without this dispatch, ad-hoc entries were silently skipped at line 48's `if (!project) continue` — the UI showed values but the server didn't save edits. This is why the read-only hack existed briefly; the proper fix is the update path. Remarks still flow into ad-hoc self-reports on update (consistent with regular self-report remark handling).
- Badge label in the submit form next to ad-hoc project names is the project type (Mandate / DDE / Pitch), not the generic word "ad-hoc". The type is already conveyed by the section header ("Added by you / teammates"); the badge slot communicates type-of-project, matching how regular projects show their stage.
- Fellow x Project timeline reads `submissions INNER JOIN cycles` alone (no `conflicts` join) because conflict resolution UPDATE-s both the VP and associate submission rows with the resolved value (`api/resolve/route.ts:56-77`). Post-resolution, submissions is single source of truth. Query filters with `or(and(fellowRecordId=X, isSelfReport=true), and(targetFellowId=X, isSelfReport=false))` — union of a fellow's own self-reports and VP projections aimed at them.
- Pure-function core for testability: `buildTimeline`, `listProjectsForFellow`, `iyOf` live in `lib/fellow-project-timeline.ts` with zero Next/DB/React imports. 18 unit tests cover IY boundaries, self-vs-projection dedup (array-order independent), IY filtering, alphabetical sort, multi-IY selection. The API route is a thin adapter that loads rows + calls these pure functions + returns JSON. Same pattern as `form-entries.ts`: if a piece of logic can be a pure function, make it one and test it exhaustively before touching the UI.
- Three-mode API (`GET /api/dashboard/fellow-project?mode=bootstrap|projects|timeline`): cascading picker pattern. Bootstrap returns all fellows + all IYs the system has seen. `projects` takes fellow + IYs (comma-separated), returns that fellow's projects within those IYs. `timeline` takes fellow + project, returns the sorted point series. Keeps each request cheap — no pre-loading everyone's entire history.
- IY (IndigoEdge fiscal year) convention: July–June. `iyOf(dateStr)` returns `year + 1` for Jul–Dec dates, `year` for Jan–Jun. IY2026 = July 2025 through June 2026. The dashboard's existing `getIyRange(iy)` uses the same convention — both should stay in sync if the fiscal calendar changes.
- **`conflicts` table now serves two shapes (`submission` and `director_flag`).** The `source` discriminator column is the gatekeeper. Submission-source rows fill `vpSubmissionId` + `associateSubmissionId` + `vpHoursPerDay` + `associateHoursPerDay` + `difference`. Director-flag rows fill `flaggedSubmissionId` + `flaggedByFellowId` + `flaggedOriginalHoursPerDay` + `proposedHoursPerDay` + `signoffId` + `resolverFellowId` + `resolverEmail`. Five existing NOT NULL constraints were dropped to allow nulls on the irrelevant side. Application code branches on `source` for insert + resolve writeback paths; tests assert both shapes round-trip cleanly.
- **Unique constraint `(cycleId, directorFellowId)` on `director_signoffs` is the idempotency anchor.** Two concurrent slice-completion checks (e.g., two near-simultaneous submissions both completing a director's slice) race on INSERT — one succeeds, one fails with a constraint violation, which the app catches and silently no-ops. Net result: exactly one signoff row, exactly one email sent, no double-fire.
- **`getDirectorSliceStatus` in-scope filter uses team membership, not submissions count.** A project is "in scope" for a director's slice iff `vpAvpIds.length + associateIds.length > 0` — having a team. Earlier draft of the spec used "≥1 submission this cycle," which would have wrongly marked a slice complete when tokens were pending (no submissions yet ≠ no team). Same fix propagated to the cycle gate (`checkAndFinalizeCycle`) and both dashboard surfaces (chip + panel) so all four consumers of "in-scope" agree.
- **`keep_original` resolution does NOT update the submission row.** It reads the submission's current `hoursPerDay` fresh and just stamps `conflicts.resolvedHoursPerDay`. This avoids accidentally reverting the submission if some other path mutated it between flag-time and resolve-time (extremely unlikely in normal operation, but the cheapest defensive posture). `use_proposed` and `custom` actions DO update the submission with the new value + recomputed `hoursPerWeek` and `autoScore`.
- **`formatDateRange` extracted from `email.ts` to `schedule.ts`** as a shared export. Pattern: any pure helper used by ≥2 modules should live in a dedicated module that doesn't import SDK clients. Same reason the original `isCycleMonday` extraction happened — keeps the function unit-testable without Resend/DB env vars.
- **`fetchEligibleFellows` and `fetchDirectors` represent disjoint sets and must not be confused.** Same Airtable Fellows List feeds both, but the helpers split the roster by intent: `fetchEligibleFellows` returns submitters (VP/AVP/Associate 1-3); `fetchDirectors` returns sign-off recipients (designation contains 'Director'). The app treats Directors and submitters as non-overlapping populations — Directors don't submit bandwidth, VPs/Associates don't receive sign-off emails. Any code that resolves a recordId to a profile must pick the helper that matches its intent. Sign-off → `fetchDirectors`. Submission validation / conflict resolution → `fetchEligibleFellows`. If both populations are needed (e.g., a unified roster view), call both and merge — don't widen either function's filter. The May 18 sign-off bug came from `createSignoffIfReady` and the cycle finalization gate calling `fetchEligibleFellows` to resolve a directorFellowId, which silently failed for every real director.

- Full self-test validated the complete workflow (cycle start through dashboard rendering). Note (historical): the original April writeback narrative format differed from manual entries; that path was retired on 2026-04-30 along with the writeback feature.
- `TEST_EMAIL_OVERRIDE` pattern works well for safe production testing. Set the env var, run the test, remove the var. No code changes needed.

### Email Delivery
- Resend Node SDK (v2+) returns `{ data, error }` instead of throwing on failure. Must destructure and check `error` explicitly, or failures are silently swallowed. The `sendEmail()` wrapper in `email.ts` handles this.
- Resend rate-limits at ~2 emails/second. Sequential sends without delays will get partially dropped. A 500ms `sleep()` between sends in `cycle.ts` and `send-reminders/route.ts` keeps us under the limit. Adds ~16 seconds to a 33-person cycle start, which is fine for a biweekly function.

### Git / Vercel deploy hygiene

- **`vercel --prod` from local uploads the working tree, including untracked files; `git push` only ships committed history.** This is the single most important deploy fact for this repo. Past `vercel --prod` deploys carried a week of uncommitted UI work (MEU removal, Fellow x Project tab, dashboard fixes) that never made it to git. The 2026-04-28 pending_projects push exposed this when the git-based build broke on a half-finished MEU type. Lesson: after any feature ships via `vercel --prod`, immediately commit the working tree and push — never let local-only state accumulate.
- **`git add -A <dir>` will sweep up untracked files in that directory tree.** The 2026-04-28 admin-removal commit used `git add -A app/src/app/api/` to stage deletions, and accidentally staged a then-untracked `dashboard/fellow-project/route.ts` whose dependency files lived outside the `-A` scope — leaving HEAD with a broken import. Going forward, prefer `git add <specific paths>` or use `git rm` + `git add` per file when there's untracked work nearby.

### Deployment
- `vercel.ts` (TypeScript config via `@vercel/config/v1`) caused silent deploy failures with empty build logs. Replaced with standard `vercel.json`. Stick with `vercel.json` for now.
- Vercel `git connect` can't parse SSH host aliases (e.g., `git@github-ie-central:`). Temporarily set remote to `https://github.com/...` URL, connect, then restore SSH remote.
- `vercel env add` via CLI reads from stdin. Use `printf` (not `echo`) to avoid trailing newline/whitespace, which Vercel rejects.
- Drizzle schema drift: `drizzle-kit push` syncs schema.ts to the DB but doesn't generate migration files. After Phase 1 of v2, prod was correct but `drizzle/` was 5 commits behind schema.ts. Run `drizzle-kit generate` alongside any schema change and commit the SQL file so the on-disk migrations stay a faithful diff record. Generated 0001_v2_schema.sql retroactively on 2026-04-20 — no-op against prod but paired with schema.ts going forward.
- Env values pulled via `vercel env pull` come back wrapped in double quotes with a literal `\n` trailing many of them (an artefact of how they were uploaded). Local test scripts need to strip both before use — a quick regex pass (`key="?value"?` → `key=value`, then trim trailing `\n`) is enough. The production runtime doesn't see this because Vercel normalizes at inject time; it only bites when you source the pulled file locally.
- **`vercel.json` `installCommand` has a hard 256-char limit; exceeding it fails the deploy at schema validation, before install runs.** This silently broke every *git-triggered* deploy for weeks — prod only advanced via manual `vercel --prebuilt` deploys, which skip vercel.json validation, so the pipeline looked "frozen" on the last manual deploy. The long-assumed "private-dep install failure" was wrong; the private dep was never the problem (the error was `installCommand should NOT be longer than 256 characters`, read straight from the Vercel deployment object). Fix: move the git-token rewrite into `app/vercel-install.sh` and point `installCommand` at it (`bash vercel-install.sh`, 22 chars). Verify a git deploy actually reaches READY — a green local `pnpm build` proves nothing about the git path. To read a failed deploy's cause without the UI: `GET https://api.vercel.com/v13/deployments/<id>` → `errorMessage`/`errorStep`.
- **SDK clients instantiated at module load make `next build` require runtime secrets at build time.** `const resend = new Resend(process.env.RESEND_API_KEY)` at the top of `email.ts` threw during `next build`'s page-data collection whenever the key was absent — breaking every preview build (preview env lacks prod secrets) and any prod build after a key lapse. Instantiate lazily (`resendClient()` on first send) so the build never depends on a runtime secret. Same principle as the DB lazy-Proxy and the `isCycleMonday`/`formatDateRange` extractions: keep import-time side-effect-free.
- Dropping a Drizzle-managed column when the new app code still reads it will crash prod. Safe order: (1) ship code that stops reading the column, (2) ship code that stops writing it, (3) update schema.ts, (4) `drizzle-kit generate`, (5) apply the migration. We did 1+2+3 in the same commit this round because the column had been write-only for days, but the rule still holds: never let the column drop land before the last reader.

### Feature Flags & Rollout

- Weekly vs biweekly cadence is a single-constant change (`CYCLE_ANCHOR` + `WEEKS_PER_CYCLE` in `schedule.ts`). Reference-date schedule math absorbs the switch — no data migration needed when changing cadence.
- `DISABLE_AIRTABLE_WRITEBACK` was a single-env-flag pause for the writeback feature (2026-04-20). Removed entirely on 2026-04-30 along with the feature. Lesson worth keeping: a single boolean env var checked at the one mutating callsite is enough to gate a whole feature path — far less code churn than threading a flag through types or branches. When the gated feature was eventually killed, the cleanup was 6 files because the flag had stayed cleanly localized.

### Source-of-truth boundaries

- **Airtable is read-only from the app's perspective** (post 2026-04-30). The app reads fellows, projects, stages, and team assignments from Airtable, but never writes back. The dashboard owns bandwidth-of-truth in Postgres (snapshots + submissions). When deciding where a new piece of data belongs: project staffing / metadata → Airtable; per-cycle bandwidth, utilization, conflicts, narratives → app DB.
- **Removing one-way writebacks simplifies the mental model dramatically.** Two-way sync between Airtable and the app DB was always a question of "which one is truth right now?" — solved at the cost of ongoing maintenance and the pause/resume gymnastics of April. Kill the writeback, pick the dashboard as truth, accept that Airtable's bandwidth fields are frozen historical text. Net code: 148 lines deleted, zero added. Future features that feel like they want a write back to Airtable should justify why the dashboard isn't the right home first.
- **`pending_projects` is the one exception to "no Airtable writes."** ie-checkin (a separate app) drains the queue and writes Airtable on BT's behalf. BT itself still doesn't touch Airtable — it just exposes the queue + lifecycle endpoints. The integration boundary is `BT_INTEGRATION_SECRET` headers on `/api/admin/pending-projects/*`. ie-checkin owns all Airtable mutation logic; BT owns lifecycle state.
- **`lib/similarity.ts` (orphaned since April 28) is now actively used again** — ie-checkin ports the same dice-coefficient function for fuzzy candidate matching against Airtable before creating stubs. Keeping orphaned utilities around can pay off when the next adjacent feature needs them.
- **The remarks endpoints extend the `pending_projects` exception, same boundary.** `/api/admin/remarks` (atomic claim + dedup) and `/api/admin/remarks/:id/processed` expose bandwidth remarks to ie-checkin under `BT_INTEGRATION_SECRET`; BT still never writes Airtable. **neon-http gotcha (cost us a prod 500):** the `drizzle-orm/neon-http` driver has NO interactive transactions and NO `SELECT … FOR UPDATE`. The first remarks claim used `db.transaction()` + `FOR UPDATE SKIP LOCKED` and 500'd on the first real call (mocked-db unit tests hid it — only live verification caught it). The fix is a single atomic `UPDATE … WHERE id IN (SELECT …) RETURNING` — statement-level row locking is race-safe without a transaction. Any future atomic claim in this app must be one statement; do not reach for `db.transaction`.

### Notion Rules Governance

- **The Notion rules sync parses ONE rule per DB row.** The `Key` property (rich_text) is the rule key; the payload is the first fenced `json` code block in the page body — the parser stops at the first block.
